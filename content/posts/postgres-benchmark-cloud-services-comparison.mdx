## "Postgres 어디 쓰세요?" 에 대한 데이터 기반 답

개발자 커뮤니티에서 가장 자주 나오는 질문 중 하나다. AWS RDS? Supabase? Neon? Railway? 의견은 많은데 데이터는 없었다.

Hacker News에서 84점을 기록한 PostgresBench가 이 질문에 답한다. 동일한 조건에서 여러 클라우드 Postgres 서비스의 성능을 **재현 가능한 방식으로** 벤치마크한 프로젝트다.

## 왜 이 벤치마크가 의미 있는가

기존 벤치마크들의 문제:
- 벤더가 직접 하면 자기한테 유리한 조건으로 테스트
- 블로거가 하면 환경 설정이 제각각
- 재현이 안 되면 검증이 불가능

PostgresBench는 **오픈소스 + 재현 가능**이 핵심이다. 누구나 같은 스크립트를 돌려서 결과를 확인할 수 있다.

## 클라우드 Postgres 서비스 선택 가이드

벤치마크 결과를 종합하면, 서비스 선택은 "성능"보다 **"상황"**에 따라 달라진다:

| 상황 | 추천 | 이유 |
|------|------|------|
| 사이드 프로젝트 | **Supabase** | 무료 티어 넉넉, Auth/Storage 통합 |
| 스타트업 MVP | **Neon** | 서버리스, 브랜칭, 자동 스케일링 |
| 프로덕션 (한국) | **AWS RDS** | 서울 리전, 안정성 검증됨 |
| 글로벌 서비스 | **CockroachDB** | 멀티 리전 분산 |
| 비용 최소화 | **Railway** | 단순 과금, 소규모에 유리 |

## 성능만으로 고르면 안 되는 이유

벤치마크에서 가장 빠른 서비스가 최선이 아닌 5가지 이유:

**1. 레이턴시는 리전이 결정한다**

서비스 자체 성능보다 서버 위치가 더 중요하다. 한국 사용자 대상이면 서울 리전이 있는 서비스가 무조건 유리하다. 아무리 빨라도 미국 서부 리전이면 100ms+ 레이턴시가 붙는다.

**2. 무료 티어의 함정**

Supabase 무료: DB 500MB, 동시 연결 제한
Neon 무료: 컴퓨트 시간 제한, 자동 서스펜드
Railway 무료: 월 $5 크레딧

무료 티어로 시작하되, **스케일 비용을 미리 계산**해야 한다.

**3. 연결 풀링**

서버리스 환경(Vercel, Cloudflare Workers)에서는 DB 연결 관리가 핵심. 연결 풀링을 기본 제공하는 서비스가 훨씬 편하다.

- Supabase: Supavisor 내장
- Neon: 연결 풀링 내장
- AWS RDS: RDS Proxy 별도 설정 필요

**4. DX(개발자 경험)**

PostgresBench가 측정 안 하는 것: 얼마나 쓰기 편한가.

- 대시보드 UI
- CLI 도구
- 마이그레이션 지원
- 로컬 개발 환경

이건 직접 써봐야 안다.

**5. 벤더 락인**

표준 PostgreSQL을 쓰면 마이그레이션이 쉽지만, 각 서비스의 독자 기능(Supabase Realtime, Neon Branching 등)을 쓰면 이전이 어려워진다.

## 실전 의사결정 플로우차트

```
Q: 한국 사용자 대상인가?

    ├─ 서버리스 필요? → Neon
    ├─ Auth/Storage 통합? → Supabase
    └─ 안정성 최우선? → AWS RDS
```
