## 핵심 요약

- Cognitive Debt(인지 부채)는 AI가 생성한 코드의 양이 개발자의 이해 속도를 초과할 때 발생하는 새로운 형태의 부채다
- Technical Debt는 "나중에 고치자"는 의식적 선택이지만, Cognitive Debt는 "이해 못 하는 코드가 쌓이는" 무의식적 누적이다
- CTO들의 공감대: 엔지니어 90%가 AI 사용을 원하지만, 코드 리뷰 자동화와 기술 건강 팀 없이는 12개월 내에 유지보수 위기가 온다

## 코드는 늘었는데 이해도는 줄었다

HN에서 39개 댓글이 달린 이 주제가 개발자들 사이에서 논쟁이 된 이유는 단순하다. 대부분이 이미 겪고 있기 때문이다. GitHub Copilot이나 Cursor로 코드를 빠르게 생성하고, 작동하니까 커밋하고, 3주 뒤에 그 코드를 수정해야 할 때 "이게 뭐 하는 코드지?"라는 상황. 이게 Cognitive Debt의 시작이다.

## Technical Debt vs Cognitive Debt — 무엇이 다른가

| 구분 | Technical Debt | Cognitive Debt |
|------|---------------|----------------|
| 원인 | 의식적 설계 타협 ("일단 하드코딩하고 나중에 리팩토링") | 무의식적 이해 부족 ("AI가 짠 코드인데 작동하니까 넘어감") |
| 인식 시점 | 부채를 만든 시점에 인지 | 수정/디버깅 시점에야 발견 |
| 누적 속도 | 개발자 생산성에 비례 | AI 코드 생성 속도에 비례 (10~50배 빠름) |
| 해결 방법 | 리팩토링 | 리팩토링 + 코드 이해 시간 투자 |
| 측정 가능성 | 비교적 쉬움 (TODO, 복잡도 지표) | 어려움 (이해도를 어떻게 측정?) |

핵심 차이는 **누적 속도**다. Technical Debt는 개발자가 코드를 직접 쓰는 속도로 쌓인다. Cognitive Debt는 AI가 코드를 생성하는 속도로 쌓인다. Copilot이 제안하는 코드를 리뷰 없이 수락하면, 하루에 수백 줄의 "이해 못 하는 코드"가 코드베이스에 들어간다.

## CTO들이 실제로 말하는 것

ShiftMag의 CTO 인터뷰에서 나온 핵심 발언들:

> "프로세스(빌드/바이 결정, 장기 유지 가치 평가)가 존재한 이유가 있습니다."

AI 코드 생성이 빨라지면서 "일단 만들고 보자"는 문화가 퍼지고 있다. 하지만 "왜 이걸 만드는가", "이걸 5년 유지할 수 있는가"라는 질문은 속도와 관계없이 필요하다.

> "엔지니어들의 정신 건강을 확인하세요."

이건 가볍게 넘길 말이 아니다. "AI가 다 해주니 편해졌다"의 이면에는 "내가 짜지 않은 코드의 버그를 내가 책임져야 하는" 스트레스가 있다. 코드 리뷰어의 인지 부하가 코드 작성자의 부하를 초과하는 역전 현상이 이미 발생하고 있다.

> "결국 토큰 소비량과 기능 가치의 예측 가능한 상태에 도달할 것 같습니다."

장기적으로는 "AI에 토큰을 얼마나 쓰고, 그 대가로 얼마나 가치 있는 기능을 만들었는가"를 측정하는 체계가 생길 거라는 전망이다.

## Cognitive Debt의 5가지 증상

자기 팀에 해당되는지 점검해볼 만한 리스트:

1. **PR 리뷰 시간이 코드 작성 시간보다 길어졌다** — AI가 코드를 빨리 생성하지만, 리뷰는 사람이 한다. 생성:리뷰 비율이 역전되면 경고 신호다.

2. **"이 코드 누가 짰지?" 질문이 늘었다** — 과거에는 git blame으로 작성자를 찾으면 그 사람에게 물어볼 수 있었다. AI가 짠 코드는 커밋한 사람도 설명 못 할 수 있다.

3. **비슷한 기능이 다른 스타일로 여러 개 존재한다** — AI는 맥락 없이 매번 새로 생성한다. 기존 유틸 함수가 있는데 비슷한 함수를 또 만드는 중복이 늘어난다.

4. **테스트 커버리지는 올라가는데 신뢰도는 떨어진다** — AI가 생성한 테스트는 "통과하는 테스트"를 만들 뿐, "의미 있는 테스트"를 만들지 않을 수 있다. 커버리지 숫자만 보면 좋아 보인다.

5. **온보딩 시간이 줄지 않는다** — 코드 생성은 빨라졌지만, 새 팀원이 코드베이스를 이해하는 시간은 오히려 늘어난다.

## 대응 전략 — 3가지 축

### 1. 코드 리뷰 자동화 (AI로 AI를 견제)

역설적이지만, AI가 만든 코드를 검증하는 데도 AI를 쓸 수 있다. 신뢰도 점수를 매겨서 고위험 코드만 사람이 집중 리뷰하는 방식이다.

- 복잡도 높은 변경 → 시니어 리뷰 필수
- 기존 패턴과 불일치 → 자동 플래그
- 테스트 없는 변경 → 머지 차단

### 2. 기술 건강 팀 (Tech Health Team)

기능 개발과 리팩토링을 동등한 업무로 인정하는 팀. "기능 출시"만 성과로 치면 Cognitive Debt는 계속 쌓인다. 주기적으로 "이해도가 낮은 코드"를 식별하고 정리하는 역할이 필요하다.

### 3. 평가 시스템 투자

모델이 바뀔 때마다 코드 품질이 변한다. GPT-4o에서 잘 되던 코드 생성이 다른 모델에서 다르게 나올 수 있다. 모델 의존성을 줄이려면 자체 평가 스위트가 있어야 한다.

## 수치로 보는 현실

| 지표 | 수치 |
|------|------|
| AI 사용 희망 엔지니어 비율 | 90% (익명 설문) |
| ROI 기대 기간 (대규모 조직) | 12개월 이내 |
| PR 리뷰 시간 증가 | 정량 데이터 아직 부족 (체감 보고만 존재) |

90%가 AI를 쓰고 싶어하지만, 그 결과물을 관리하는 체계는 대부분 없다. 이 간극이 Cognitive Debt의 온상이다.

## 판단

Cognitive Debt라는 용어가 정착될지는 모르겠지만, 현상 자체는 이미 진행 중이다. "AI가 짠 코드인데 작동하니까 넘어갔다"는 경험이 있는 개발자라면, 이미 인지 부채를 지고 있는 것이다.

해법은 AI를 안 쓰는 게 아니다. AI 코드 생성 속도에 맞는 검증/이해 체계를 갖추는 것이다. 코드를 빨리 만드는 것보다, 만든 코드를 빨리 이해하는 능력이 2026년 개발자의 핵심 역량이 되고 있다.
