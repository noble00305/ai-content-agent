## "돌아가니까 됐잖아"가 가장 위험한 말이다

Hacker News에 올라온 "When I reject AI code even if it works"라는 글이 개발자들 사이에서 뜨거운 공감을 얻고 있다. AI가 생성한 코드가 테스트를 통과하고, 빌드도 되고, 잘 돌아가는데도 리젝하는 경우가 있다는 것이다.

왜? **코드는 "돌아가는 것"이 전부가 아니기 때문이다.**

## AI 코드를 거부해야 하는 7가지 상황

### 1. 이해할 수 없는 코드

AI가 생성한 코드를 팀원 누구도 설명할 수 없다면, 그건 기술 부채다. 버그가 나면? 성능을 개선해야 하면? 아무도 손댈 수 없는 코드가 된다.

**체크**: "이 코드가 왜 이렇게 작동하는지 설명할 수 있는가?"

### 2. 과도한 추상화

AI는 "깔끔한 코드"를 만들려다 3줄이면 될 걸 인터페이스 + 팩토리 + 전략 패턴으로 30줄로 만든다. 지금 필요 없는 유연성을 위해 복잡도를 올리는 건 나쁜 트레이드오프다.

**체크**: "이 추상화가 지금 당장 필요한가, 아니면 '나중에 필요할 수도'인가?"

### 3. 보안 취약점이 숨어있을 때

AI는 SQL 인젝션, XSS, 하드코딩된 시크릿 같은 보안 이슈를 자연스럽게 생성한다. 코드가 돌아가는 것과 안전한 것은 완전히 다른 문제다.

**체크리스트:**
- [ ] 사용자 입력을 직접 쿼리에 넣고 있지 않은가?
- [ ] 민감 정보가 하드코딩되어 있지 않은가?
- [ ] 에러 메시지에 내부 구조가 노출되지 않는가?
- [ ] 인증/인가 체크가 빠져있지 않은가?

### 4. 테스트 없이 복잡한 로직

AI가 알고리즘을 생성했는데 테스트 코드가 없다? 지금 돌아가는 건 우연일 수 있다. 엣지 케이스에서 터지는 건 시간 문제다.

**체크**: "이 코드에 대한 테스트가 있는가? 엣지 케이스를 커버하는가?"

### 5. 프로젝트 컨벤션을 무시할 때

팀이 camelCase를 쓰는데 AI가 snake_case로 생성하거나, 특정 라이브러리 대신 다른 걸 쓰거나. 일관성이 깨지면 유지보수 비용이 기하급수적으로 늘어난다.

**체크**: "기존 코드베이스의 패턴/컨벤션을 따르고 있는가?"

### 6. 라이선스 문제 가능성

AI가 오픈소스 코드를 학습 데이터에서 거의 그대로 가져왔을 가능성이 있다. GPL 코드가 섞이면 법적 리스크가 생긴다.

**체크**: "이 코드가 특정 오픈소스 프로젝트의 코드와 유사하지 않은가?"

### 7. 성능이 숨겨진 폭탄일 때

AI가 만든 코드가 10건에서는 잘 돌아가는데, 10만 건에서는 O(n²)이라 터진다. AI는 "돌아가는 코드"를 만들지, "스케일하는 코드"를 만들지는 않는다.

**체크**: "데이터가 10배, 100배 늘어나면 이 코드가 버틸 수 있는가?"

## 그래서 AI 코드를 어떻게 써야 하나

AI 코드를 무조건 거부하라는 게 아니다. **검증 프로세스**를 거치라는 것이다.

**실용적 워크플로우:**

```
AI 코드 생성
    ↓
1단계: 이해 가능한가? → 못하면 리젝
    ↓
2단계: 보안 체크 → 취약점 있으면 수정 또는 리젝
    ↓
3단계: 컨벤션 맞는가? → 안 맞으면 수정
    ↓
4단계: 테스트 있는가? → 없으면 작성
    ↓
5단계: 코드 리뷰 → 사람이 최종 판단
    ↓
머지
```

핵심은 **AI를 "1차 초안 작성자"로 쓰되, 최종 판단은 사람이 하는 것**이다. AI가 80%를 만들어주고, 사람이 20%를 검증하고 다듬는 구조가 가장 효율적이다.
