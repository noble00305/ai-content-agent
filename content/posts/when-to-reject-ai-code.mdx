## 핵심 요약

- AI 코드가 "돌아간다" ≠ "써도 된다"
- 보안, 가독성, 컨벤션, 성능 4가지 축으로 검증 필요

## 거부 기준 7가지

### 1. 팀 누구도 이 코드를 설명할 수 없다

AI가 만든 코드를 아무도 설명 못 하면 기술 부채. 버그 나면 손댈 수 없는 코드가 된다.

**체크:** "이 코드가 왜 이렇게 동작하는지 설명할 수 있는가?"

### 2. 3줄이면 될 걸 30줄로 만들었다

AI는 "깔끔한 코드"를 만들려다 인터페이스 + 팩토리 + 전략 패턴을 과도하게 쓴다.

**체크:** "이 추상화가 지금 당장 필요한가?"

### 3. 보안 취약점이 숨어있다

AI는 SQL 인젝션, XSS, 하드코딩 시크릿을 자연스럽게 생성한다.

**체크리스트:**
- 사용자 입력을 직접 쿼리에 넣고 있지 않은가?
- 민감 정보가 하드코딩되어 있지 않은가?
- 에러 메시지에 내부 구조가 노출되지 않는가?
- 인증/인가 체크가 빠져있지 않은가?

### 4. 테스트 없이 복잡한 로직

테스트 없는 AI 생성 알고리즘은 지금 돌아가는 게 우연일 수 있다.

**체크:** "테스트가 있는가? 엣지 케이스를 커버하는가?"

### 5. 프로젝트 컨벤션을 무시한다

camelCase 프로젝트에 snake_case, 특정 라이브러리 대신 다른 거 사용 등.

**체크:** "기존 코드베이스의 패턴을 따르고 있는가?"

### 6. 라이선스 문제 가능성

AI가 오픈소스 코드를 거의 그대로 가져왔을 수 있다. GPL 코드 섞이면 법적 리스크.

**체크:** "특정 오픈소스 프로젝트 코드와 유사하지 않은가?"

### 7. 10건에서는 돌아가는데 10만 건에서 터진다

AI는 "돌아가는 코드"를 만들지, "스케일하는 코드"를 만들지는 않는다.

**체크:** "데이터 10배, 100배 늘어나면 이 코드가 버틸 수 있는가?"

## AI 코드 리뷰 워크플로우

```
AI 코드 생성
  → 이해 가능? (No → 리젝)
  → 보안 체크 (취약점 → 수정/리젝)
  → 컨벤션 맞음? (No → 수정)
  → 테스트 있음? (No → 작성)
  → 사람 최종 판단
  → 머지
```

AI를 "1차 초안 작성자"로 쓰되, **최종 판단은 사람이 하는 것**이 원칙.
