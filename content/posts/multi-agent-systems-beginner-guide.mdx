## 멀티 에이전트 시스템이란

단일 AI 에이전트는 하나의 LLM이 모든 작업을 처리한다. 멀티 에이전트 시스템은 여러 에이전트가 각자 역할을 맡아 협업하는 구조다.

비유하자면:

- **단일 에이전트** = 1인 스타트업. 기획, 개발, 디자인, 마케팅을 한 명이 다 함
- **멀티 에이전트** = 팀. 기획자, 개발자, 디자이너, 마케터가 각자 전문 영역 담당

실제 구성 예시:

```
┌─────────────────────────────────────────────┐
│              오케스트레이터 (관리자)            │
│         작업 분배 + 결과 취합 + 품질 판단        │
└──────┬──────────┬──────────┬────────────────┘
       │          │          │
  ┌────▼───┐ ┌───▼────┐ ┌──▼─────┐
  │ 코드    │ │ 테스트  │ │ 보안   │
  │ 생성기  │ │ 작성기  │ │ 스캐너 │
  └────────┘ └────────┘ └────────┘
       │          │          │
       └──────────┴──────────┘
              결과 취합
```

## 왜 지금 멀티 에이전트인가

### 1. LLM 성능이 에이전트를 가능하게 하는 수준에 도달

2023년까지 LLM은 도구 호출(function calling)이 불안정했다. 2025~2026년 모델들은 도구 호출 정확도가 95% 이상으로 올라왔고, 긴 컨텍스트에서도 지시를 잘 따른다. 에이전트의 기본 전제인 "지시를 정확히 이행"이 가능해진 것이다.

### 2. 단일 에이전트의 한계

하나의 에이전트에게 "코드 작성 + 테스트 + 보안 검토 + 문서화"를 모두 시키면:
- 컨텍스트 윈도우를 빠르게 소진
- 역할 간 간섭 (보안 규칙이 코드 생성 스타일을 제약)
- 실패 시 전체 작업 중단

역할을 분리하면 각 에이전트가 자기 영역에 집중할 수 있고, 하나가 실패해도 다른 에이전트에 영향이 적다.

### 3. 기업 수요 폭증

단순 챗봇을 넘어 실제 업무 자동화에 AI를 적용하려는 기업이 늘면서, 복잡한 워크플로우를 처리할 수 있는 멀티 에이전트에 대한 관심과 수요가 빠르게 커지고 있다. LangChain, CrewAI, Microsoft AutoGen 등 전용 프레임워크가 잇따라 등장한 것이 그 방증이다.

### LangGraph

```python
# LangGraph 기본 구조 (개념 코드)
from langgraph.graph import StateGraph

workflow = StateGraph(State)
workflow.add_node("researcher", research_agent)
workflow.add_node("writer", writing_agent)
workflow.add_node("reviewer", review_agent)

workflow.add_edge("researcher", "writer")
workflow.add_edge("writer", "reviewer")
workflow.add_conditional_edges("reviewer", quality_check,
    {"pass": END, "fail": "writer"})
```

그래프 구조라서 조건 분기, 루프, 병렬 실행 등 복잡한 흐름을 표현할 수 있다. 대신 학습 곡선이 있다.

### CrewAI

```python
# CrewAI 기본 구조 (개념 코드)
from crewai import Agent, Task, Crew

researcher = Agent(role="리서처", goal="최신 트렌드 조사")
writer = Agent(role="작가", goal="조사 결과를 글로 정리")

task1 = Task(description="AI 트렌드 조사", agent=researcher)
task2 = Task(description="조사 결과로 글 작성", agent=writer)

crew = Crew(agents=[researcher, writer], tasks=[task1, task2])
crew.kickoff()
```

직관적인 역할/태스크 구조. 빠르게 프로토타입을 만들 수 있다.

## 시작하기: 단계별 가이드

### Step 1: 단일 에이전트부터 시작

멀티 에이전트를 바로 구축하기 전에, 하나의 에이전트가 도구를 사용하여 작업을 수행하는 구조부터 만든다. LLM + 도구 호출이 안정적으로 동작하는지 확인하는 단계다.

### Step 2: 2개 에이전트로 확장

가장 간단한 멀티 에이전트 구조: **생성자 + 검증자**. 에이전트 A가 결과물을 만들고, 에이전트 B가 검증한다. 이것만으로도 단일 에이전트 대비 품질이 올라간다.

### Step 3: 오케스트레이터 추가

에이전트가 3개 이상이 되면 관리자 역할의 오케스트레이터가 필요하다. 작업을 분배하고, 에이전트 간 결과를 전달하고, 최종 품질을 판단한다.

### Step 4: 실패 처리 설계

멀티 에이전트에서 가장 중요한 것은 실패 처리다. 에이전트 하나가 실패했을 때 전체가 멈추지 않도록 재시도, 대체 에이전트, 타임아웃 등을 설계해야 한다.

## 주의사항

- **오버엔지니어링 경계**: 단순한 작업에 멀티 에이전트를 적용하면 오히려 복잡성만 늘어난다. 단일 에이전트로 충분한지 먼저 판단할 것
- **비용 관리**: 에이전트 수 × LLM 호출 횟수 = 비용. 에이전트가 불필요하게 대화를 주고받으면 비용이 급증
- **디버깅 난이도**: 에이전트 간 상호작용이 복잡해지면 어디서 문제가 생겼는지 파악하기 어렵다. 로깅을 철저히 할 것
- **환각 전파**: 에이전트 A의 환각이 에이전트 B의 입력이 되면 오류가 증폭된다. 각 단계에서 검증 로직 필요
