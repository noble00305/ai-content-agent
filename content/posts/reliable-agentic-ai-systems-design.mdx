## PRINCE 시스템 — 제약 규제 문서를 처리하는 에이전트

Bayer의 PRINCE는 제약 규제 문서(수만 페이지 PDF)를 검색하고 분석하는 에이전트 시스템이다. 단순 검색(Search)에서 출발해, 자연어 질의응답(Ask), 최종적으로 규제 문서 자동 작성(Do)까지 3단계로 진화했다.

핵심은 "의료 분야에서 틀리면 안 된다"는 제약이 설계 전체를 지배한다는 점이다. 이 제약이 만들어낸 패턴들이 범용적으로 적용 가능하다.

---

## 설계 원칙 1: Context Engineering — 정보를 분리하라

모든 정보를 한 프롬프트에 때려넣는 건 가장 흔한 실수다. PRINCE는 역할별로 문맥을 엄격히 분리한다.

| 에이전트 역할 | 받는 정보 | 안 받는 정보 |
|-------------|----------|------------|
| Think & Plan | 사용자 질문 + 계획 지침 | 원본 문서 |
| Researcher | 검색 쿼리 + 문서 청크 | 다른 에이전트의 추론 과정 |
| Writer | 검증된 데이터 + 작성 규칙 | 검색 과정의 중간 결과 |

왜 이렇게 하는가? LLM은 관련 없는 정보가 많을수록 핵심을 놓친다. 프롬프트가 길어질수록 성능이 떨어지는 건 벤치마크로 증명된 사실이다. 역할별 문맥 분리는 이 문제를 구조적으로 해결한다.

---

## 설계 원칙 2: 3단계 반영 루프

PRINCE의 가장 인상적인 부분이다. 출력을 바로 사용자에게 보내지 않고, 3번 걸러낸다.

**1단계 — Process Reflection (방향 점검)**
Think & Plan 에이전트가 "이 질문에 대해 이 방향으로 조사하겠다"는 계획을 세우면, 반영 모듈이 "이 계획이 질문의 의도에 맞는가?"를 검증한다.

**2단계 — Data Reflection (데이터 충분성)**
Researcher가 수집한 데이터를 별도 에이전트가 평가한다. "이 데이터만으로 답변할 수 있는가? 빠진 정보는 없는가?" 부족하면 추가 검색을 지시한다.

**3단계 — Draft Reflection (답변 완성도)**
Writer가 작성한 초안을 검증한다. 인용 출처가 정확한지, 논리적 비약이 없는지 확인한 후에야 사용자에게 전달한다.

이 구조의 비용은? LLM 호출이 최소 3배 늘어난다. 하지만 "한 번 틀린 답변의 비용"이 "LLM 호출 3배의 비용"보다 크다면 합리적인 트레이드오프다.

## 설계 원칙 3: RAG 파이프라인의 구체적 수치

PRINCE의 RAG 설정은 구체적 수치가 공개되어 참고 가치가 높다.

| 파라미터 | 값 | 이유 |
|---------|---|------|
| 초기 검색 청크 수 | ~20개 | 재현율 확보 |
| 재랭킹 후 선택 | 7개 | 정밀도와 컨텍스트 길이 균형 |
| 하이브리드 검색 가중치 | 시맨틱 0.7 / 키워드 0.3 | 의미 검색 우선, 정확한 용어 매칭 보조 |
| 쿼리 확장 | 5개 변형 | 의료 용어 동의어 커버 (piloerection → goosebumps) |

특히 쿼리 확장이 흥미롭다. 사용자가 "piloerection, ataxia"를 검색하면 시스템이 "goosebumps, lack of coordination"으로 자동 확장한다. 도메인 특화 동의어 확장은 범용 RAG에서도 적용 가능한 패턴이다.

## 설계 원칙 4: 장애를 전제한 설계

PRINCE는 "LLM은 반드시 실패한다"를 전제로 설계됐다.

**자동 복구 메커니즘:**
- SQL 쿼리 실패 시 3회까지 자동 수정 시도
- LLM 응답 실패 시 다른 제공자의 모델로 자동 페일오버
- 사용자 재시도 시 이미 완료된 단계는 건너뜀 (PostgreSQL에 상태 저장)

**검증 체계:**
- 일일 배치 평가 — Faithfulness(사실 충실도), Answer Relevancy(답변 관련성), Context Relevancy(문맥 관련성) 정량 측정
- 주요 변경 시 전체 데이터셋 평가 실행

이 부분이 대부분의 에이전트 프로젝트에서 빠지는 곳이다. "잘 되면 좋고, 안 되면 다시 돌리자"가 아니라, 장애를 시스템의 일부로 포함시켜야 한다.

## 자체 에이전트에 적용할 수 있는 체크리스트

| 항목 | 질문 | PRINCE의 답 |
|------|------|------------|
| 문맥 분리 | 각 에이전트가 필요 이상의 정보를 받고 있지 않은가? | 역할별 프롬프트 분리 |
| 반영 루프 | 출력을 검증하는 단계가 있는가? | 3단계 (방향/데이터/초안) |
| 장애 복구 | LLM 실패 시 자동으로 대응하는가? | 3회 재시도 + 모델 페일오버 |
| 상태 저장 | 중간에 실패해도 처음부터 다시 시작하지 않는가? | PostgreSQL 상태 관리 |
| 정량 평가 | 에이전트 성능을 수치로 측정하고 있는가? | 일일 배치 + 변경 시 전체 평가 |

## 판단

Martin Fowler 팀 글의 가치는 "이론"이 아니라 "실제 프로덕션 시스템의 구체적 수치와 의사결정"을 공개했다는 점이다. RAG 청크 수, 가중치, 반영 루프 횟수 — 이런 구체적인 숫자는 자기 시스템을 설계할 때 출발점으로 쓸 수 있다.

다만 PRINCE는 제약 규제라는 고위험 도메인이다. 모든 에이전트에 3단계 반영 루프가 필요한 건 아니다. 중요한 건 "신뢰성 비용을 어디에 얼마나 투자할 것인가"를 의식적으로 결정하는 것이다.
