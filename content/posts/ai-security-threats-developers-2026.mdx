## 3. LiteLLM CVE-2026-42271 — 실제 사례

LiteLLM은 여러 LLM API를 통합 관리하는 프록시 도구로, 스타트업과 개발팀에서 많이 쓴다. 이 도구에서 발견된 취약점 내용:

| 항목 | 내용 |
|------|------|
| CVE 번호 | CVE-2026-42271 |
| 유형 | Command Injection |
| 심각도 | Critical |
| 인증 필요 여부 | **불필요 (Unauthenticated)** |
| 영향 | 원격 코드 실행 (RCE) |
| 영향 범위 | LiteLLM 프록시를 외부에 노출한 모든 환경 |

**대응 방법**:
- LiteLLM 최신 버전으로 즉시 업데이트
- 프록시 서버를 퍼블릭 인터넷에 직접 노출하지 않기
- 인증 레이어 (API 키, OAuth) 반드시 설정

이 사례가 보여주는 패턴: AI 도구 자체가 공격 표면이 된다. AI 스택을 도입할수록 보안 점검 범위도 넓어진다.

## 4. AI 지원 공격이 표준이 되고 있다

AI를 활용한 공격은 더 이상 실험 단계가 아니다.

| 공격 유형 | AI 활용 방식 | 위험도 |
|----------|-------------|--------|
| 피싱 | 타겟 맞춤 이메일 자동 생성 | 매우 높음 |
| 코드 취약점 탐색 | 오픈소스 코드 자동 분석 | 높음 |
| 소셜 엔지니어링 | 딥페이크 음성/영상 | 매우 높음 |
| 자격증명 스터핑 | 패턴 기반 비밀번호 추측 | 중간 |
| 공급망 공격 | 악성 패키지 자동 생성/배포 | 매우 높음 |

특히 공급망 공격이 위험하다. npm, PyPI 같은 패키지 레지스트리에 정상처럼 보이는 악성 패키지를 AI로 대량 생성할 수 있다. 타이포스쿼팅(이름이 비슷한 패키지)도 AI가 자동으로 만든다.

|----------|------|----------|

| 에이전트 하이재킹 | AI 에이전트의 도구 호출 조작 | 도구 권한 최소화, 샌드박싱 |
| 악성 컨텍스트 주입 | RAG 소스에 악성 지시 삽입 | 소스 검증, 컨텍스트 격리 |
