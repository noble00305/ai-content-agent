## 국가별 도입률 — 격차가 크다

### 선두 그룹 (60%+)

| 국가 | IPv6 도입률 | 주요 동인 |
|------|-----------|----------|
| 인도 | ~70% | Reliance Jio가 IPv6를 기본 프로토콜로 채택 |
| 베트남 | ~60% | 모바일 우선 인프라, 정부 주도 전환 정책 |
| 사우디아라비아 | ~60% | 신규 인프라 구축 시 IPv6 우선 |

공통점: **모바일 네트워크 중심으로 성장한 국가**가 빠르다. 기존 IPv4 인프라에 투자한 비용이 적을수록 전환이 쉽다.

### 중간 그룹 (30~50%)

미국(~45%), 독일(~50%), 일본(~45%). 대형 ISP와 클라우드 사업자가 듀얼 스택을 운영하면서 점진적으로 전환 중.

### 후발 그룹 (<30%)

중국(~25%), 러시아, 아프리카 다수 국가. Carrier-Grade NAT로 IPv4를 연장하는 전략을 택한 곳이 많다.

## 듀얼 스택의 현실 — "두 프로토콜 세계"

IPv6가 50%라는 건 IPv4가 사라졌다는 뜻이 아니다. 현재 인터넷은 두 프로토콜이 공존하는 상태다. 이게 개발자에게 복잡성을 더한다.

**현재 인프라 스택:**
```
사용자 → [IPv6 or IPv4] → ISP (NAT/CGNAT) → CDN (듀얼 스택) → 서버
```

- NAT: 사설 IPv4 주소를 공인 IP로 변환
- CGNAT (Carrier-Grade NAT): ISP가 여러 사용자를 하나의 공인 IPv4로 묶음
- 듀얼 스택: IPv4와 IPv6를 동시에 지원

CGNAT 환경에서 발생하는 실무 문제:
- IP 기반 rate limiting이 부정확해짐 (수천 명이 같은 IP)
- 지리 기반 서비스가 틀릴 수 있음
- WebSocket 연결이 불안정해지는 사례 보고

## 개발자 체크리스트 — 지금 확인할 것

| 항목 | 확인 방법 | 왜 중요한가 |
|------|----------|-----------|
| 서버가 IPv6를 지원하는가 | `curl -6 https://yourdomain.com` | 트래픽 절반이 IPv6인데 서버가 IPv4만 지원하면 CDN/프록시에 의존 |
| DNS에 AAAA 레코드가 있는가 | `dig AAAA yourdomain.com` | AAAA 없으면 IPv6 클라이언트가 직접 접근 불가 |
| 로깅에 IPv6 주소를 저장하는가 | 로그 포맷 확인 | IPv6 주소는 128비트, 기존 VARCHAR(15) 필드로는 부족 |
| Rate limiting이 IPv6를 고려하는가 | /64 서브넷 단위로 제한하는지 | IPv6는 /128(단일 IP)로 제한하면 우회가 쉬움, /64 단위가 표준 |
| 방화벽 규칙에 IPv6가 포함되는가 | `ip6tables -L` 또는 클라우드 보안 그룹 | IPv4만 차단하고 IPv6를 열어두는 실수가 흔함 |
| CI/CD에서 IPv6를 테스트하는가 | GitHub Actions는 IPv6 미지원이 기본 | 프로덕션과 CI 환경의 네트워크 차이 |

## 2030년 전망

APNIC 데이터 기준 2018년 이후 3년에 약 10%p씩 선형 성장하고 있다. 이 추세가 유지되면:

- 2027년: ~55% (IPv6가 확실한 다수)
- 2030년: ~65% (IPv4-only 서비스가 문제가 되기 시작)
- 2033년+: ~75% (IPv4 레거시 취급)

다만 IPv4가 완전히 사라지는 건 수십 년 후일 수 있다. CGNAT 기술이 IPv4의 수명을 계속 연장하고 있기 때문이다.

## 판단

IPv6 50% 돌파는 "축하할 이벤트"라기보다 "더 이상 무시할 수 없는 전환점"이다. 특히 글로벌 서비스를 운영하는 개발자라면, 인도·동남아 사용자의 다수가 이미 IPv6로 접속하고 있다는 사실을 인지해야 한다.

당장 전체 인프라를 IPv6로 전환할 필요는 없다. 하지만 위 체크리스트의 항목들 — AAAA 레코드, IPv6 로깅, rate limiting — 은 지금 점검해도 이르지 않다. "나중에 하면 되지"가 통했던 시간은 50%와 함께 끝났다.
