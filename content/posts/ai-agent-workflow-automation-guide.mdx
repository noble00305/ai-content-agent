## 매크로와 에이전트의 차이부터 구분하자

Zapier로 "Gmail 첨부파일 → Google Drive 저장"을 설정하는 건 매크로다. 규칙이 정해져 있고, 예외가 생기면 멈춘다. AI 에이전트는 다르다. "이 이메일이 중요한지 판단하고, 중요하면 요약해서 슬랙에 보내고, 아니면 아카이브해라"처럼 **판단이 포함된 작업**을 처리한다.

이 차이를 이해하면, 어떤 업무에 에이전트가 필요하고 어떤 업무에는 단순 자동화로 충분한지 구분할 수 있다.

## 워크플로우 1: 이메일 분류 + 요약 + 라우팅

**문제**: 하루 50~100통의 이메일 중 실제로 중요한 건 5~10통. 나머지를 걸러내는 데 매일 30분.

**에이전트 구성:**

| 단계 | 에이전트 역할 | 도구 |
|------|------------|------|
| 수신 | 새 이메일 감지 | Gmail API / Zapier trigger |
| 분류 | 긴급/일반/스팸 판단 | Claude API / GPT API |
| 요약 | 긴급 메일 3줄 요약 생성 | LLM |
| 라우팅 | 슬랙 채널로 요약 전송 | Slack API |
| 아카이브 | 스팸/불필요 메일 자동 정리 | Gmail API |

**구현 팁:**
- 분류 정확도를 높이려면 최근 2주간 직접 분류한 이메일을 few-shot 예시로 제공
- "긴급" 판단 기준을 명시적으로 정의: 발신자 도메인, 키워드, 금액 언급 등
- 오분류 대비: 아카이브 전 24시간 유예 기간 설정

## 워크플로우 2: 데이터 수집 + 정리 + 스프레드시트 업데이트

**문제**: 여러 소스에서 데이터를 모아 스프레드시트에 정리하는 주간 작업. 매번 2시간.

**에이전트 구성:**

| 단계 | 역할 | 도구 |
|------|------|------|
| 수집 | 웹 데이터, API, RSS 수집 | Python requests / Playwright |
| 정제 | 형식 통일, 중복 제거, 결측값 처리 | LLM + pandas |
| 검증 | 이상값 플래그 | LLM 판단 ("이 수치가 정상 범위인가?") |
| 기록 | Google Sheets 업데이트 | Sheets API |
| 알림 | 이상값 발견 시 슬랙 알림 | Slack API |

**실전 예시 — 경쟁사 가격 모니터링:**
```
매주 월요일 09:00
→ 경쟁사 5곳 가격 페이지 크롤링
→ 가격 변동 감지 (±5% 이상)
→ 비교표 생성 → Google Sheets 업데이트
→ 변동 있으면 슬랙 알림
```

## 워크플로우 3: 일일/주간 리포트 자동 생성

**문제**: 대시보드 데이터를 보고 요약 리포트를 쓰는 작업. 매일 20분 또는 매주 1시간.

**에이전트 구성:**

| 단계 | 역할 | 도구 |
|------|------|------|
| 데이터 추출 | GA, 슬랙, Jira 등에서 지표 수집 | 각 API |
| 분석 | 전일/전주 대비 변화 계산, 이상 패턴 감지 | LLM + 계산 |
| 작성 | 한국어 리포트 생성 (3~5문단) | LLM |
| 배포 | 이메일 또는 슬랙으로 발송 | 이메일/Slack API |

**리포트 품질을 높이는 프롬프트 구조:**
```
역할: 주간 성과 분석가
입력: {이번 주 지표 JSON}
규칙:
- 숫자만 나열하지 말고 "왜 이렇게 변했는가" 가설을 1개 이상 제시
- 전주 대비 변화가 ±10% 이상인 항목만 하이라이트
- 마지막에 "다음 주 주목할 것" 1줄 추가
```

## 워크플로우 4: 시스템 모니터링 + 알림 + 초기 대응

**문제**: 서버/서비스 이상 감지 후 인간이 확인하고 판단하기까지 시간 소요.

**에이전트 구성:**

| 단계 | 역할 | 도구 |
|------|------|------|
| 감지 | 에러율, 응답시간, CPU 등 모니터링 | Prometheus / CloudWatch |
| 판단 | 심각도 분류 (정보/경고/긴급) | LLM (과거 인시던트 패턴 참조) |
| 초기 대응 | 긴급 시 자동 스케일링 또는 롤백 트리거 | 스크립트 실행 |
| 알림 | 심각도별 다른 채널로 알림 | PagerDuty / Slack |
| 기록 | 인시던트 타임라인 자동 작성 | LLM |

**주의**: 자동 롤백 같은 파괴적 조치는 반드시 사람 승인 단계를 넣어야 한다. "80% 자동 + 20% 사람" 원칙이 가장 중요한 영역.

| 공유 | 요약 + 액션 아이템을 슬랙에 공유 | Slack API |

## "80/20 원칙" — 완벽한 자동화를 목표로 하지 마라

가장 흔한 실패: 100% 자동화를 목표로 엣지 케이스 처리에 시간을 쏟다가 포기. 현실적인 접근은 이렇다.

| 자동화 비율 | 사람 역할 | 적합한 업무 |
|-----------|----------|-----------|
| 100% | 없음 | 로그 수집, 백업 같은 기계적 작업만 |
| 80% | 최종 확인, 예외 처리 | 이메일 분류, 리포트 생성, 데이터 정리 |
| 50% | 핵심 판단 | 고객 응대, 채용 스크리닝 |
| 20% | 에이전트는 보조만 | 전략 결정, 협상, 크리에이티브 |

대부분의 반복 업무는 80% 자동화로 충분하다. 나머지 20%를 자동화하는 데 드는 비용(시간, 오류 리스크)이 직접 하는 비용보다 클 때가 많다.

## 판단

AI 에이전트 자동화의 핵심은 "무엇을 자동화하느냐"가 아니라 "어디까지 자동화하느냐"를 결정하는 것이다. 위 5가지 워크플로우 중 자기 업무에 해당하는 게 있다면, 100%를 목표로 시스템을 설계하지 말고, 가장 반복적인 80%만 먼저 자동화해보는 걸 권한다. 그것만으로도 매일 30분~1시간을 아낄 수 있다.
