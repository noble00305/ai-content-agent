## TypeScript가 사실상 표준이 된 이유

| 요인 | 설명 |
|------|------|
| AI 도구 호환성 | AI 코딩 도구가 TypeScript에 최적화 — 타입 정보가 더 정확한 코드 생성을 유도 |
| 프론트+백 통합 | Next.js, Nuxt 등 풀스택 프레임워크의 기본 언어 |
| 타입 안전성 | 대규모 코드베이스에서 리팩토링 안정성 |
| 생태계 규모 | npm 패키지 대부분이 TypeScript 지원 |
| 채용 시장 | TypeScript 요구 공고가 순수 JS 공고를 넘어섬 |

순수 JavaScript로 새 프로젝트를 시작하는 건 이제 **의도적인 선택**이 아니면 레거시로 분류된다. AI 도구를 활용할 때도 TypeScript의 타입 정보가 코드 생성 품질을 크게 올린다.

AI 에이전트들도 이 메타 프레임워크를 기본으로 사용한다. v0은 Next.js를, Bolt.new은 여러 프레임워크를 지원하지만 Next.js가 기본값이다. **프레임워크를 아는 것**이 곧 **AI 도구를 잘 쓰는 것**과 직결된다.

## 새로 필요한 역량 5가지

### 1. 프롬프트 엔지니어링

단순히 "이거 만들어줘"가 아니라, AI가 정확한 결과를 내도록 **맥락, 제약조건, 출력 형식**을 설계하는 능력.

| 수준 | 프롬프트 예시 | 결과 품질 |
|------|------------|----------|
| 초급 | "로그인 기능 만들어줘" | 기본적이지만 보안 취약 |
| 중급 | "Next.js App Router에서 NextAuth.js로 Google OAuth 구현, 세션 관리 포함" | 쓸 수 있는 수준 |
| 고급 | "기존 미들웨어 패턴에 맞춰, rate limiting 포함, 에러 핸들링은 프로젝트 컨벤션 따를 것" | 프로덕션 수준 |

### 2. 에이전트 오케스트레이션

여러 AI 에이전트에게 역할을 나누고, 결과를 조합하는 능력. 예: 코드 생성 에이전트 + 테스트 작성 에이전트 + 보안 검사 에이전트를 파이프라인으로 연결.

### 3. AI 코드 리뷰

AI가 생성한 코드에서 **보안 취약점, 성능 문제, 아키텍처 결함**을 잡아내는 능력. AI가 코드를 많이 쓸수록, 리뷰의 가치가 올라간다.

### 4. 시스템 설계

구현 디테일은 AI에 맡기더라도, **전체 아키텍처를 설계하는 능력**은 사람의 몫이다. 마이크로서비스 구조, 데이터 모델링, API 설계 같은 상위 의사결정.

### 5. 도메인 지식

코드는 AI가 쓰더라도, **"무엇을 만들어야 하는지"**를 결정하는 건 도메인을 이해하는 사람이다. 금융, 의료, 교육 등 분야별 지식의 가치가 오히려 올라간다.

## 정리

개발자 역할 변화의 방향:

1. **코드 작성 → 시스템 설계 + AI 조율**로 무게중심 이동
2. **TypeScript + 메타 프레임워크**가 기본기 — 여기서 시작
3. **프롬프트 엔지니어링, 에이전트 오케스트레이션**이 새로운 핵심 역량
4. **도메인 지식**의 가치가 상대적으로 올라감
5. 코드를 짜는 속도보다 **올바른 것을 만드는 판단력**이 차별화 요소

변화 속도가 빠르다고 불안해할 필요는 없지만, 무시하면 뒤처진다. 지금 쓰고 있는 AI 도구의 출력을 **이해하고 검증하는 습관**이 가장 현실적인 첫 걸음이다.
